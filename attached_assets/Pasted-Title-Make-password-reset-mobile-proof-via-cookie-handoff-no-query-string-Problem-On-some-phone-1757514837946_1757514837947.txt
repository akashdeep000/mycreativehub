Title: Make password reset mobile-proof via cookie handoff (no query string)

Problem: On some phones, the email client or redirect chain drops the ?token=… so /reset-password can’t read it and we get errors. We need a flow that doesn’t rely on the query string.

Please implement exactly:

Create a short server route that captures the token and sets a cookie.

Route: GET /rp/:token

Behavior:

app.get('/rp/:token', (req, res) => {
  const token = req.params.token;
  if (!token) return res.status(400).send('Missing token');

  res.cookie('reset_token', token, {
    httpOnly: true,
    secure: true,
    sameSite: 'lax',
    path: '/',            // or '/reset-password' if you prefer
    maxAge: 1000 * 60 * 15 // 15 minutes
  });

  // Now send the user to the clean SPA route without any token in the URL
  res.redirect(302, '/reset-password');
});


Change the email link to use the short path (no query):

const BASE = process.env.APP_BASE_URL; // https://mycreativehub.app
const link = `${BASE}/rp/${token}`;
// Use `link` for the button and the plain-text fallback


On the client reset page, don’t read the token from the URL.

Either:

When the user submits the new password, POST to /api/auth/reset-password with only the new password. The backend reads req.cookies.reset_token and performs the reset; or

Provide a tiny API /api/auth/reset-token that returns the token from the cookie so the client can show better messages.

After a successful reset, clear the cookie on the server:

res.clearCookie('reset_token', { path: '/', httpOnly: true, secure: true, sameSite: 'lax' });


Keep the SPA fallback and ESM-safe imports.

For non-API GETs, always serve index.html (no redirects, no throwing).

Ensure the server uses ESM imports:

import path from 'node:path';
import { fileURLToPath } from 'node:url';
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);


If you force HTTPS/host, preserve the path with req.originalUrl.

Acceptance tests

From desktop: visit an email link like https://mycreativehub.app/rp/<token> → lands on /reset-password and works.

From iPhone Mail: tap the button → lands on /reset-password and works (no token in the URL is needed).

Reusing the same link after success is rejected.

Logs show the initial /rp/<token> request, then a 302 to /reset-password.
